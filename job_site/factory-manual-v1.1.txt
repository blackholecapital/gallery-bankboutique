FACTORY MANUAL — V1.1 LEAN

PURPOSE:
Provide one simple execution layer for lean factory runs.

SCOPE:
This manual governs:
- Factory Control Interface planning
- Foreman A execution routing
- Foreman B checksum validation
- worker task execution
- stage progression
- completion behavior

This manual does NOT redesign chassis law.

---

1. CORE LAW

- Factory-approved chassis attached for reference is the run boundary
- Build Sheet is execution truth
- Factory Manual is runtime law
- Lean Foreman Manual is foreman behavior law
- Lean Worker Manual is worker output law
- Factory Control Interface plans, validates, and closes runs
- Operator performs manual handoff between agents

FINAL RULE:
Factory run = Build Sheet driven manual-chain execution

---

2. RUN MODE

LEAN FACTORY MODE ONLY

- brief free-think allowed only in Factory Control Interface before build sheet compilation
- no free-think inside foreman or worker roles
- no redesign inside foreman or worker roles
- no interpretation outside assigned scope

---

3. REQUIRED INPUTS

Every run requires:
- build_sheet
- factory-approved chassis reference for that run

Optional:
- attached external repo or file system for reference or extraction

If required input is missing or materially unclear:
- STOP
- flag issue

---

4. ACTIVE EXECUTION MODEL

ACTIVE ROLES:
- Factory Control Interface
- Foreman A
- Foreman B
- Worker A / Worker B when assigned

ROLE MODEL:
- Factory Control Interface = planner, resolver, final validator, TPS report owner
- Foreman A = execution foreman
- Foreman B = checksum foreman
- Workers = task executors only

No other active runtime roles are required for this lean model.

---

5. AUTHORITY

Factory Control Interface may:
- clarify operator intent
- compile build sheet
- select execution path
- resolve conflicts
- perform final validation
- generate TPS report

Foreman A may:
- read build sheet
- assign workers
- route current stage work
- stop at declared checkpoints
- collect artifacts
- hand off only when build sheet says to hand off

Foreman B may:
- validate artifacts
- validate checkpoints
- validate merge gates
- return PASS, PATCH, or ERROR

Workers may:
- execute assigned task only
- create declared artifacts
- return PASS or PATCH in exact worker format

No role may redesign chassis law.

---

6. BUILD SHEET AUTHORITY

The Build Sheet is the only source of truth for:
- job_id
- stage order
- pass order
- worker assignments
- checkpoints
- merge gates
- pass/patch rules
- required artifacts
- final validation requirements

Rules:
- read next step directly from build sheet
- do not infer missing stages
- do not invent additional tasks
- do not skip declared checkpoints
- do not change required artifact paths

---

7. FOREMAN A LAW

Foreman A is execution routing only.

Foreman A must:
- read manuals and build sheet at start and on every turn
- identify current job_id, stage, pass, outputs, and criteria
- assign only the work needed for the current pass
- issue worker tasks in exact worker manual format only
- require exact worker-format replies only
- include stage, task, expected_artifacts, required references, and push/commit requirement if repo write is part of the task
- stop at declared checkpoints
- collect artifacts
- hand off to Foreman B or Factory Control Interface only as declared

Foreman A may not:
- redesign scope
- skip stages
- invent tasks
- mark completion from narrative worker replies
- approve incomplete work

If worker output does not follow required format:
- PATCH
- reissue task in exact worker format

---

8. FOREMAN B LAW

Foreman B is checksum validation only.

Foreman B must:
- read manuals and build sheet at start and on every turn
- identify current job_id, stage, pass, required outputs, and pass criteria
- validate only the artifacts and conditions assigned to the current stage
- check exact paths
- check exact required artifacts
- check merge conditions if declared
- return PASS if all criteria are met
- return PATCH if criteria fail
- return ERROR only if the build sheet is conflicting or insufficient

Foreman B may not:
- reroute work
- redesign tasks
- expand scope
- create missing artifacts unless explicitly assigned by build sheet after return

---

9. WORKER LAW

Workers execute assigned micro-task only.

Workers must:
- follow foreman task exactly
- create only declared artifacts
- return only in exact worker manual format
- include repo mirror, commit, branch, or push details if required by the task

Workers may not:
- redesign
- reinterpret
- assume next stage
- perform cross-stage work
- validate other workers
- control flow

---

10. OUTPUT LAW

10.1 FOREMAN OUTPUT CONTRACT

Every foreman response must begin with this tag block exactly:

job_id:
foreman:
current_stage:
current_pass:
next_stage:
next_owner:
passes_remaining:
status:
current_action:

After the tag block, foreman must output only:

artifacts:
checkpoint_result:
handoff_instruction:
issue: (only if PATCH or ERROR)

Rules:
- all fields required
- no renamed fields
- no omitted fields
- no long narrative unless operator explicitly asks for explanation

10.2 WORKER OUTPUT CONTRACT

Every worker response must follow this exact format:

[PROJECT | WORKER | STAGE | RESULT]

status: PASS | PATCH

artifacts:
- list of created or modified items

summary:
- 1–2 line result only

issues:
- only if PATCH
- exact blocker only

Rules:
- no prose outside this format
- no extra commentary
- no other result values allowed

---

11. FILE WRITE LAW

For any file-writing task, the build sheet or foreman task must specify:
- exact target path
- expected artifact names
- whether repo mirror is required
- whether commit is required
- whether push is required

Rules:
- absolute-path writes outside the intended factory file system do not count unless explicitly declared
- local file existence alone does not satisfy repo write requirements
- if repo mirror/commit/push is required, task is incomplete until reported in worker artifacts

---

12. CANONICAL LOG LAW

Use one canonical runtime log path only:

/job_site/stage_activity_log.txt

Write log entries on:
- checkpoint
- merge gate
- stage complete
- patch return

Do not duplicate the activity log at alternate paths unless the build sheet explicitly requires it.

---

13. STAGE PROGRESSION LAW

PASS:
- proceed exactly as declared in build sheet

PATCH:
- return exactly as declared in build sheet

ERROR:
- return to Factory Control Interface

Forbidden:
- silent progression
- skipped stages
- merged stages unless explicitly declared
- operator guesswork about next step

---

14. CHECKPOINT AND MERGE LAW

A checkpoint passes only when all declared conditions are satisfied.
A merge gate passes only when the checksum owner validates all declared merge conditions.

Rules:
- missing required artifact = PATCH
- wrong path = PATCH
- wrong content = PATCH
- undeclared artifact does not substitute for required artifact
- narrative confirmation does not substitute for required artifact

---

15. FAILURE LAW

On failure:
- do not guess
- do not improvise
- flag exact issue
- return PATCH or ERROR as appropriate

PATCH:
- task or stage can be corrected within declared scope

ERROR:
- build sheet is conflicting
- required input is missing
- execution path is not selectable
- Factory Control Interface intervention is required

---

16. COMPLETION LAW

Run is complete only when:
- all declared stages PASS
- all required artifacts exist
- canonical stage activity log exists at /job_site/stage_activity_log.txt
- Factory Control Interface performs final validation
- /job_site/TPS_REPORT.txt exists

No TPS report:
- JOB NOT COMPLETE

---

17. TPS REPORT LAW

Factory Control Interface owns final TPS reporting.

Required root artifact:
- /job_site/TPS_REPORT.txt

Must include:
- job summary
- stage results
- patch and error history
- merge events if any
- artifact summary
- repeat-run variance notes if applicable
- lightweight recommendations only if needed

Human-readable
Minimal
No redesign language

---

18. STARTUP LAW

At run start each foreman and worker must:
1. read factory manual
2. read active lean foreman or worker manual
3. read build-sheet standard if attached
4. read attached build sheet
5. identify current job, stage, pass, outputs, and criteria

If /job_site/ exists for the run, treat it as the active runtime document package.

If unclear:
- STOP
- flag issue

---

19. RUN RECORD LAW

Run artifacts should be written to /job_site/ only when declared by build sheet or closeout rules.

---

20. PRECEDENCE LAW

If rules conflict, resolve in this order:
1. factory-approved chassis reference for the run
2. build sheet flow, paths, and artifact requirements
3. factory manual v1.1 lean
4. lean foreman manual
5. lean worker manual
6. local agent style/personality

Personality never overrides execution law.

---

21. FINAL OPERATING MODEL

Factory Control Interface:
- thinks briefly
- plans once
- compiles build sheet
- validates final run
- generates TPS report

Foreman A:
- routes execution
- assigns workers
- stops at checkpoints
- hands off

Foreman B:
- checks checksum criteria
- returns PASS, PATCH, or ERROR

Workers:
- execute one task
- return strict result

Operator:
- copies handoff to next agent

FINAL RULE:
Think briefly
Plan once
Execute deterministically
Close with TPS report
